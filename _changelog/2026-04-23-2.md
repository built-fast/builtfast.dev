Environment responses now expose a companion `provisioning_step_label` field and the `provisioning_step` enum has been expanded to report fine-grained provisioning progress.

<!--more-->

### API Improvements

**New `provisioning_step_label` field**

- Every environment response now includes `provisioning_step_label` alongside `provisioning_step` -- a human-readable label (for example `Queued`, `Deploying`, `Waiting for custom domain DNS`, `Complete`) suitable for UI display without requiring clients to maintain their own mapping.

**Expanded `provisioning_step` values**

New granular steps have been added to surface each stage of the provisioning pipeline:

- `pending` -- Queued
- `creating_ymir_project` -- Creating Ymir project
- `creating_ymir_environment` -- Creating Ymir environment
- `creating_database` -- Creating database
- `creating_redis_user` -- Creating Redis user
- `creating_cdn` -- Creating CDN (production only)
- `creating_dns` -- Creating DNS records
- `generating_salts` -- Generating WordPress salts

Several existing step labels have been clarified:

| Step | Previous label | New label |
| --- | --- | --- |
| `requesting_cert` | Requesting AWS ACM certificate | Requesting SSL certificate |
| `waiting_cert` | Waiting for certificate validation | Waiting for SSL certificate |
| `deploying` | Deploying to Ymir | Deploying |
| `configuring_dns` | Configuring DNS records | Verifying DNS propagation |
| `provisioning_cdn_ssl` | Requesting Let's Encrypt certificate | Provisioning CDN SSL |

**`POST /vector/sites/{site}/environments/{environment}/deployments`**

- The `include_database` parameter now defaults to `true` (previously `false`). Deployments will include database changes by default; pass `include_database: false` explicitly to skip.

See the [Vector Pro API documentation](/api/) for full details.
